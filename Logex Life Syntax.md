# Syntax for life

Let's create a good and bad character to show all combinations of logic:
- Jane - a good character.
- Ogre - a bad character.

This is mixed Laegna-Latin, because it does not use Laegna's own truth value symbols, but creatively uses Latin for Laegna semantics!

# Syntax

Let's use not or not not (checkmark) on left and right side - left side is local (causal, state) and right side is global (goal-based, goal). This means Logex is state-goal-based automation derived from Laegna Logecs - Laegna Logic and Math. "-ex" suffix in Laegna, in general, could mean automation especially if "cs" of science is replaced with "ex", sounding so similar that "U"-pronouncing rules are applied: if it's not so different, it might be hard to notice the difference in laegna, such as "U" and upside-down-U are pronounced similarly, such as some similar-meaning characters could look the same on first sight, or some fonts: probability is considered as meaningful and communicative.

## Jane is doing fine

The following are equivalent expressions to show Jane is good and should be doin good.

```prolog
Jane.
```

```prolog
✓Jane.
```

```prolog
✓Jane✓.
```

Syntactic sugar: ✓ could be opposite of not, emphasizing the yes - it's often confusing are you saying "not not x" to say it's not set that it's not true, or you say "x" where it's set that x is true - checkmark can avoid this, giving literal differences to thosem, Logecs-important factors as binary. ✓ could be used at empty places, to emphasize lack of ¬ visually, or to show it's logic.

```prolog
Jane = E.
```

This state Goal and State of Jane are True.
- Jane is doing well - True is an average of True, plus or minus.
- Jane should be doing well.
- Jane could be doing better, and this is what the Logex Machine will be now looking in combinations.
- A and E are goal to do well.
- O and E are state of doing well.

## Ogre is fine but shouldn't be

Goal is set *after* the identifier (each sentence means the same):

```prolog
Ogre¬.
```

```prolog
✓Ogre¬.
```

```prolog
Ogre = O.
```

Here, going good would be bad..

This state Goal of Orge is False and State is True - O is local, and collapses in infinity.
- Ogre is doing well - True is an average of True, plus or minus.
- Ogre should be going bad.
- Ogre could be doing better, and this is what the Logex Machine will be now looking in combinations.
- I and O are goal to do bad.
- O and E are state of going well.

# Jane helps Mary

Here, Jane helps Mary and should do so:

```prolog
Jane => Mary.
```

```prolog
(Jane => Mary) = E.
```

## Mary helps back

```prolog
Jane <=> Mary.
```

"<=>" uses symmetric "<=" and "=>", because the scope is singleton and they equal in same scope, whichever the real scope is.

```prolog
(Jane <=> Mary.)✓
```

It's here checked good, not assumed good.
